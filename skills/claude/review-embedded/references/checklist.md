# 审查清单（详细版）

每类只报告**发现的实际问题**，无问题的类别跳过不提。

---

## 1. 内存管理

### 1.1 动态分配
- [ ] `malloc` / `calloc` / `realloc` 后是否检查返回值（NULL 判断）
- [ ] 每条代码路径（含错误分支、提前 return）是否都有对应的 `free`
- [ ] 循环内重复分配是否在循环出口/错误分支正确释放
- [ ] 释放后是否置 `NULL` / `-1`（防 double-free，本项目约定）

### 1.2 mmap
- [ ] `mmap` 返回值是否检查 `MAP_FAILED`（不是 NULL）
- [ ] 是否与 `munmap` 成对，且 munmap 的 size 与 mmap 一致

### 1.3 DMA / 平台缓冲区
- [ ] `dma_buf_alloc` 是否配对 `dma_buf_free`
- [ ] `mpp_buffer_get` 是否配对 `mpp_buffer_put`
- [ ] MPP/RGA 任务对象是否在所有路径上销毁
- [ ] 错误路径是否遗漏了 `release_buffer_handle` 等清理调用

---

## 2. 缓冲区安全

- [ ] `memcpy` / `memset` 的 size 参数是否可能超过目标缓冲区大小
- [ ] **非 POD 对象（含虚函数表、构造函数的 C++ 对象）禁止使用 `memcpy` / `memset`**，会绕过构造/析构，破坏对象状态
- [ ] `sprintf` → 应替换为 `snprintf`（长度限制）
- [ ] 固定大小字符数组是否有越界写入风险；数组下标计算是否考虑负数情况
- [ ] `strcpy` / `strcat` 是否有长度保护（应改用 `strncpy` / `strncat`）
- [ ] `gets` → 禁止使用，改用 `fgets`
- [ ] `sscanf` / `fscanf` 中 `%s` 是否指定宽度限制（如 `%255s`），未指定宽度的 `%s` 等同于 `gets`
- [ ] 尺寸计算 `width * height * bpp` 是否可能整数溢出（应先转 `size_t` / `uint64_t`）
- [ ] 数组下标来自外部输入或循环计数时，是否在使用前验证上下界（同时检查负数下标）
- [ ] 非 null 终止的字符串（如从网络/协议解析的 JSON 字段）是否使用了 `strncpy` 并手动添加 `\0`

---

## 3. 资源泄漏

- [ ] `open` / `socket` 在**所有路径**（正常流 + 错误流）上是否 `close`
- [ ] `ioctl` 失败后是否清理了已获取的资源
- [ ] `goto cleanup` / 统一出口是否覆盖了所有已分配资源
- [ ] 错误返回前是否遗漏了 `release_xxx`、`destroy_xxx` 等清理调用
- [ ] **`pthread_attr_t` 使用后是否调用 `pthread_attr_destroy`**（本项目多处遗漏）
- [ ] **初始化函数中顺序分配多个资源时，中间步骤失败是否回滚已分配资源**（应使用 `goto cleanup` 或封装 `destroy` 函数）
- [ ] V4L2 MMAP 缓冲区：是否执行 `munmap` + `VIDIOC_REQBUFS(0)` 完整释放
- [ ] 多线程场景：线程退出时是否释放线程局部资源

---

## 4. 线程安全

- [ ] 全局变量 / 共享状态的读写是否有互斥锁（`pthread_mutex`）保护
- [ ] 读多写少的场景是否考虑读写锁（`pthread_rwlock`）
- [ ] **互斥锁加锁后，函数的所有退出路径（含 `return`、`break`、`goto`）是否都有对应的 `unlock`**
- [ ] `sem_post` / `sem_wait` 调用是否正确配对，无死锁路径
- [ ] `pthread_create` 后句柄是否保存，以便后续 `join` 或 `detach`
- [ ] 信号处理函数中是否调用了非 async-signal-safe 函数（`malloc`、`printf` 等）
- [ ] 条件变量是否在 while 循环中等待（避免虚假唤醒）

---

## 5. 硬件相关

- [ ] DMA 缓冲区在 CPU 读写**前后**是否调用 `dma_sync_*`（cache 一致性）
- [ ] `ioctl` 调用前结构体是否 `memset` 清零（防止内核读到脏字段）
- [ ] 设备文件（`/dev/xxx`）打开前是否检查存在性（`stat` 或权限）
- [ ] fd 有效性检查：本项目以 `fd < 0` 或 `fd == 0` 表示无效，是否一致
- [ ] 硬件寄存器读写是否使用 `volatile` 或 `ioread32` / `iowrite32`

---

## 6. 错误处理与代码正确性

- [ ] 系统调用、平台 API（`ioctl`、`mpp_*`、`rga_*`）返回值是否被检查
- [ ] **非 void 函数是否在所有路径上都有 `return` 语句**（编译器不保证报错，实际返回垃圾值）
- [ ] **`malloc` / `calloc` / 关键函数返回值是否在使用前判 NULL**，避免空指针解引用
- [ ] **先解引用、后判 NULL 的逻辑顺序错误**：确保 NULL 检查在使用之前
- [ ] **局部变量（尤其结构体字段、时间戳、返回值 `ret`）是否初始化**后再使用
- [ ] **无符号变量与 0 比较 `>= 0` 或 `< 0` 结果恒真/恒假**，应改用有符号类型或删除多余判断
- [ ] C++ 类若持有裸指针成员，是否遵循**三法则（Rule of Three）**：析构函数、拷贝构造、赋值运算符同时定义，防止浅拷贝导致 double-free
- [ ] 错误路径是否有日志输出（`SLOGI` / `logger_error` / `fprintf(stderr, ...)`）
- [ ] 部分初始化失败时是否清理了已成功初始化的部分（避免半初始化对象）
- [ ] 错误码是否正确透传给调用者（不要吃掉错误码）

---

## 7. 栈使用（仅 full 模式）

- [ ] 函数内是否有大的局部数组（嵌入式线程栈通常 4–64 KB，单个数组 > 1 KB 需关注）
- [ ] 是否有深层递归调用风险（嵌入式环境栈溢出无保护）
- [ ] 大结构体是否改为堆分配或静态分配

---

| 数组下标越界（含负数下标）| 🔴 |
